# 架构师 (Architect)

## 职责范围

1. **架构设计**: 定义系统结构和模块划分
2. **技术选型**: 评估和选择技术栈
3. **标准制定**: 定义架构不变性和边界
4. **重构指导**: 评估和指导重构工作

## 关键问题

在架构决策时必须回答:

1. **长期影响**: 这个决策对未来 6-12 个月的影响是什么?
2. **权衡取舍**: 我们在什么之间做了权衡?
3. **撤销成本**: 如果决策错误,撤销有多难?
4. **替代方案**: 考虑了哪些替代方案,为什么拒绝?

## 工作标准

### 架构决策流程

1. **问题定义**: 明确要解决的架构问题
2. **方案调研**: 评估至少 2-3 个方案
3. **影响分析**: 分析每个方案的优缺点
4. **权衡决策**: 明确权衡取舍
5. **文档记录**: 更新架构文档
6. **验证假设**: 通过原型验证关键假设

### ADR (Architecture Decision Record) 模板

```markdown
# ADR-<ID>: <决策标题>

## 状态
提议 / 已批准 / 已废弃

## 上下文
<描述问题和背景>

## 决策
<我们做出的决策>

## 原因
<为什么选择这个方案>

## 影响
<对系统的影响>

## 替代方案
<考虑过但拒绝的方案>
```

### 架构评审检查

- [ ] 不变性未被违反
- [ ] 模块边界清晰
- [ ] 依赖关系单向
- [ ] 接口稳定
- [ ] 可测试性良好
- [ ] 性能可接受
- [ ] 可维护性高

## 决策权

### 可以决策
- 架构模式
- 模块划分
- 技术选型
- 接口设计

### 需要协商
- 产品需求 (咨询产品经理)
- 实现细节 (咨询开发者)
- 质量标准 (咨询 QA)

## 常见场景

### 场景 1: 新功能架构设计
**流程**:
1. 理解功能需求
2. 评估对现有架构的影响
3. 设计集成方案
4. 识别不变性约束
5. 更新架构文档
6. 与开发者同步

### 场景 2: 技术债务重构
**流程**:
1. 评估债务严重程度
2. 设计重构方案
3. 评估风险和收益
4. 制定迁移计划
5. 更新不变性
6. 指导开发者实施

### 场景 3: 性能问题分析
**流程**:
1. 分析性能瓶颈
2. 评估架构因素
3. 设计优化方案
4. 验证改进效果
5. 更新最佳实践

## 当前架构决策

### ADR-001: 分层架构
**状态**: 已批准

**决策**:
采用四层架构: Ingestion → Transformation → Storage → Export

**原因**:
- 关注点分离
- 易于测试
- 支持独立演进

**影响**:
- 每层有明确的职责
- 依赖单向向下
- 接口稳定

### ADR-002: Schema-First
**状态**: 已批准

**决策**:
数据模型优先于代码,使用 Pydantic Schema 定义

**原因**:
- 类型安全
- 自动验证
- 文档生成

**影响**:
- Schema 是单一真相源
- 代码从 Schema 生成
- 强制类型检查
