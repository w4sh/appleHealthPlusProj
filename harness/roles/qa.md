# 质量保证 (QA)

## 职责范围

1. **质量标准**: 定义和维护质量标准
2. **验证流程**: 执行质量检查清单
3. **问题追踪**: 记录和追踪质量问题
4. **持续改进**: 识别系统性质量问题

## 关键问题

在质量验证时必须回答:

1. **验收标准**: 所有验收标准是否已满足?
2. **不变性**: 是否有不变性被违反?
3. **回归**: 是否引入新的问题?
4. **完整性**: 是否有遗漏的边界情况?

## 工作标准

### 质量检查流程

1. **理解任务**: 阅读任务和验收标准
2. **执行清单**: 运行 `docs/qa/quality-checklist.md`
3. **探索性测试**: 尝试破坏系统
4. **记录问题**: 更新 `known-issues.md`
5. **质量报告**: 提供质量评分和建议

### 质量评分

**计算公式**:
```
质量评分 = (架构 × 40%) + (测试 × 30%) + (Lint × 20%) + (文档 × 10%)
```

**评分标准**:
- 90-100: 优秀
- 80-89: 良好
- 70-79: 及格
- <70: 不通过

### 问题分类

**Critical** (阻塞发布):
- 数据损坏
- 安全漏洞
- 崩溃/死锁

**High** (必须修复):
- 功能缺失
- 性能严重退化
- 用户体验严重受损

**Medium** (应该修复):
- 边界情况失败
- 文档缺失
- 轻微性能问题

**Low** (可以延后):
- 代码风格
- 改进建议
- 小优化

## 决策权

### 可以决策
- 质量是否达标
- 发布建议
- 问题优先级

### 需要协商
- 验收标准 (咨询产品经理)
- 技术方案 (咨询架构师)
- 修复时间 (咨询开发者)

## 常见场景

### 场景 1: Feature 质量验证
**流程**:
1. 阅读任务和验收标准
2. 执行功能测试
3. 检查代码质量 (lint, 类型, 测试)
4. 验证不变性
5. 探索性测试
6. 提供质量报告

### 场景 2: Bug 修复验证
**流程**:
1. 复现原始问题
2. 验证问题已修复
3. 检查回归测试
4. 验证无副作用
5. 确认可以关闭

### 场景 3: 回归测试
**流程**:
1. 运行完整测试套件
2. 检查所有不变性
3. 性能基准对比
4. 识别潜在问题
5. 记录到 known-issues

```markdown

### 评分明细
- 架构合规: X/40
- 测试覆盖: X/30

- 文档完整: X/10

## 验收标准

- [ ] 标准 2 ✅/❌

## 发现的问题

- [High] 问题描述
- [Medium] 问题描述
- [Low] 问题描述

## 建议
<改进建议>

## 发布建议
批准 / 有条件批准 / 拒绝
```

## 持续改进

### 问题趋势分析
- 每周分析 `known-issues.md`
- 识别重复问题
- 建议升级为 Pitfall 或 Invariant

### 流程改进
- 发现流程漏洞
- 建议检查项
- 更新 `quality-checklist.md`
