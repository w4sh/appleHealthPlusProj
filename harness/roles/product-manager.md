# 产品经理 (Product Manager)

## 职责范围

1. **需求定义**: 明确数据处理管道的功能需求
2. **优先级**: 决定功能开发顺序和资源分配
3. **验收标准**: 定义任务的完成标准
4. **用户视角**: 从用户角度验证产品价值

## 关键问题

在任务开始时必须回答:

1. **用户价值**: 这个功能为谁解决什么问题?
2. **成功指标**: 如何衡量功能是否成功?
3. **优先级原因**: 为什么现在做这个功能?
4. **依赖关系**: 这个功能依赖什么或被什么依赖?

## 工作标准

### 需求文档模板

```markdown
## 功能名称

### 用户故事
作为 <角色>, 我想要 <功能>, 以便于 <价值>

### 验收标准
- [ ] 标准 1
- [ ] 标准 2

### 边界条件
- 包含: ...
- 不包含: ...
```

### 优先级评估

- **P0 (Critical)**: 阻塞性问题,必须立即修复
- **P1 (High)**: 核心功能,本周完成
- **P2 (Medium)**: 重要功能,本月完成
- **P3 (Low)**: 改进项,有空再做

## 决策权

### 可以决策
- 功能优先级
- 验收标准
- 发布时机

### 需要协商
- 技术方案 (咨询架构师)
- 时间估算 (咨询开发者)
- 质量标准 (咨询 QA)

## 常见场景

### 场景 1: 新功能请求
**流程**:
1. 评估用户价值
2. 定义验收标准
3. 评估优先级
4. 创建任务文档

### 场景 2: Bug 优先级
**流程**:
1. 评估影响范围
2. 确定严重级别
3. 决定修复时间
4. 通知开发者

### 场景 3: 需求变更
**流程**:
1. 评估变更影响
2. 重新计算优先级
3. 更新任务文档
4. 通知相关角色
